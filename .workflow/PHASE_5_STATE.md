### 5.3 Diagnostics Endpoint (Complete ✅)

**File**: `backend/src/me4brain/api/routes/diagnostics.py`

Endpoint: `GET /v1/diagnostics/llm-chain`

Response includes:
- **config** - Current LLM configuration (model_routing, providers, base URLs)
- **health** - Health check results for each provider:
  - `ollama`: healthy status, error, latency_ms, available_models
  - `lmstudio`: healthy status, error, latency_ms
- **resolution** - Model resolution chain result:
  - `ok`: Whether resolution succeeded
  - `client`: LLM client type (NanoGPTClient, etc.)
  - `model`: Selected model name
  - `error`: Error message if resolution failed
- **recommendation** - Actionable recommendation based on health status

Recommendations generated by `_generate_recommendation()`:
- ✅ OK: Ollama healthy → "✅ OK: Ollama is healthy and ready with model 'llama2'"
- ⚠️ DEGRADED: Ollama down, LM Studio up → "⚠️ DEGRADED: Using LM Studio fallback..."
- 🔴 CRITICAL: All down → "🔴 CRITICAL: No local LLM available. Start Ollama with..."

### Integration Points

**Prometheus Scraping**:
- Metrics use standard `prometheus_client` format
- Can be scraped at `/metrics` endpoint (default FastAPI location)
- Labels enable filtering by:
  - `method`: llm, fallback_keyword, fallback_heuristic
  - `error_type`: timeout, connection, parse, validation
  - `reason`: timeout, error, low_confidence
  - `has_context`: true, false

**Dashboard Integration**:
- Diagnostics endpoint provides real-time health status
- Can be queried directly: `curl http://localhost:8089/v1/diagnostics/llm-chain`
- Response includes actionable recommendations for operators

**Logging Integration**:
- Structured logs include metrics data (latency_seconds, domains, confidence)
- Error logs include error_type for tracking specific failure modes
- All events tagged with query preview for debugging
